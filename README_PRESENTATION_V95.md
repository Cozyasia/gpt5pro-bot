## Какие файлы заменить

В корне репозитория:

1. Заменить `main.py` файлом из этого комплекта.
2. Заменить или добавить `presentation_studio.py` рядом с `main.py`.
3. Заменить `requirements.txt`.

Остальные действующие файлы проекта — `secret_loader.py`, `runway_official.py`, `engine.py`, `premium.html`, `runtime.txt`, база `/data/subs.db` и Render Disk — не удалять и не заменять этим комплектом.

## Что исправлено

- Убрано ложное сообщение «Упс, произошла ошибка» после успешно созданной структуры.
- Ошибки этапов презентации теперь обрабатываются локально, а состояние проекта сохраняется.
- Главный бриф сохраняется целиком и остаётся единственным источником фактов проекта.
- Точное название бренда извлекается отдельно и проверяется; слова «бренд», «бренда», «компания», «товар», «услуга» не принимаются как название.
- Добавлен выбор логотипа: автоматически по главному брифу, по отдельному накопительному брифу, загрузить готовый или продолжить без логотипа.
- Дополнения к брифу логотипа, изображениям, стилю и палитре можно вносить несколько раз — предыдущие пожелания не стираются.
- Генератор логотипа создаёт знак без текста, а точное название накладывается программно.
- Автоматические изображения создаются в горизонтальном формате для презентации и привязываются к конкретным слайдам без смещения при сбое отдельной генерации.
- Исключены случайные абстрактные заглушки: при недоступности провайдера используется аккуратный текстовый макет, а не нерелевантная картинка.
- Число слайдов соблюдается точно; PDF и PPTX строятся из одной структуры.
- Цены, продуктовая линейка и контакты извлекаются из главного брифа и возвращаются в соответствующие слайды.
- Проекты сохраняются в SQLite и на Render Disk, поэтому перезапуск сервиса не должен уничтожать текущий этап.

## Переменные Render

Новые переменные не обязательны. При `DB_PATH=/data/subs.db` мастер автоматически использует `/data/presentation_studio`.

Для ручной настройки значения приведены в `PRESENTATION_ENV_V95.txt`.

## Деплой

1. Сделать резервную копию `/data/subs.db`.
2. Загрузить три обязательных файла в корень репозитория.
3. Выполнить commit и push.
4. В Render запустить `Manual Deploy` → `Clear build cache & deploy`.
5. Не удалять подключённый Persistent Disk.

## Проверка после запуска

В Telegram:

1. Выполнить `/version` — должна отображаться версия `v95-presentation-studio-fix-2026-07-14`.
2. Выполнить `/diag_presentation`.
3. Открыть `💼 Работа/Бизнес` → `📊 Презентация PDF + PPTX`.
4. Отправить полный тестовый бриф.
5. Утвердить структуру и убедиться, что дополнительного сообщения «Упс» нет.
6. Проверить автоматический логотип и режим отдельного брифа.
7. Несколько раз добавить пожелания к логотипу и изображениям.
8. Собрать PDF и PPTX и проверить одинаковое число слайдов.

## Локальная проверка

```bash
python -m py_compile main.py presentation_studio.py presentation_smoke_test.py
python presentation_smoke_test.py
```

Smoke-тест не вызывает реальные production API и не списывает кредиты. Он проверяет парсер бренда, 12-слайдовую структуру, сохранение проекта и локальную сборку PDF/PPTX.
